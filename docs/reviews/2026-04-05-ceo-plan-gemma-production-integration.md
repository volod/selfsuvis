## Vision

### 10x Check

The 10x version isn't "Gemma added to the demo" — it's a production-grade outdoor scene intelligence engine with a **single LLM backbone** (Gemma) that does captioning, structured extraction, and anomaly explanation, while DINOv3 handles retrieval (where it's empirically better). The edge deployment story: a 7MB EfficientViT model distilled from Gemma's language-grounded embeddings runs on a Jetson in 8ms/frame — no cloud, no sidecar, fully autonomous. Every mission produces richer captions than any existing drone/rover software. The robot advisory API returns not just "similar frames" but "last 3 visits to this GPS waypoint had wet roads and 3+ trucks, current frame caption: dry road, clear."

---

## Critical Empirical Findings (from system audit — drive scope decisions)

1. **SSL fine-tuning shows zero improvement** on test data (0 frames delta, R@1 identical). Likely a dataset issue (58 frames, visually homogeneous aerial footage). MUST be validated on diverse data before Phase 2 investment.
2. **Gemma embeddings less discriminative than DINOv3** (0.9914 vs 0.9225 mean cosine sim). User opted to keep Gemma as Phase 3 teacher (language-grounded embeddings have theoretical value for distillation), but Phase 2 SSL gate must pass first. If gate fails: distill from DINOv3 directly (DINOv3 → EfficientViT-S1, ~1.5h CC, same edge goal).
3. **Demo pipeline speed**: 42 minutes for 58 frames. Gemma API is ~16.6s/frame. Phase 1 must include async/chunked batch captioning. Target: ≤5 min for 58 frames.

## Accepted Scope

- `pipeline/indexer.py`: wire `GEMMA_API_URL` as captioning option; Florence as fallback. Implement chunked async batch (N=3 frames/chunk, 50s timeout, 1 retry per chunk). Target throughput: ≤5 min for 58 frames.
- `scripts/validate_ssl_improvement.py`: multi-video SSL eval harness. "Diverse" must include ALL of: (1) a daylight video, (2) a low-light/overcast video, (3) a video with ≥3 distinct GPS waypoints (real field footage only — synthetic/test data excluded). ≥200 frames each. Measures ΔR@1 pre/post SSL.
- `pipeline/qwen_model.py` → Gemma API replacement (after Phase 2 quality validation). Gate: Gemma JSON validity ≥ 0.82 AND vehicle count accuracy ≥ 0.70 AND ≥50 test frames (not 20 — Codex identified 20-frame test as too weak). If gate fails (<0.82 or <0.70): keep Qwen; no gray zone.
- `pipeline/ssl_finetune.py`: `GemmaSSLFinetuner` class — CONDITIONAL: build only if SSL gate passes (ΔR@1 > +0.02 on ≥2 of 3 test videos). If gate fails: proceed to EfficientViT-S1 distillation from DINOv3 teacher instead.
- `pipeline/distill.py`: `GemmaVisionTeacher` wrapper (`nn.Module` around GemmaEmbedder vision encoder) + Stage 0→1 distillation. Projection head: `Linear(384→1152)` — student (384-dim ViT-S/14) projected to match Gemma vision encoder output (1152-dim SigLIP ViT-L). RKD-DA loss from `pipeline/distill.py` existing implementation. Optional `lambda_caption_anchor=0.5` loss: `cosine(projected_student, CLIP_text(caption))`. RKD-D uses `pipeline/distill.py` — no new library required.
- `models/efficientvit_model.py`: new file — EfficientViT-S1 loader via `timm` (`timm.create_model("efficientvit_b1", pretrained=True)`). **DINOv3→EfficientViT-S1 distillation ships as the Phase 2 baseline** — RKD-D only (no RKD-A), 384-dim, no SSL gate required. Gemma teacher is a Phase 3 upgrade path, not the starting point. The baseline lets us measure whether Gemma teacher adds value over DINOv3 teacher.
- `pipeline/demo/`: refactored subpackage. `pipeline/demo_runner.py` becomes thin shim importing `from pipeline.demo.runner import run_demo` for backward compatibility. Module split: `steps_embed.py`, `steps_caption.py`, `steps_ssl.py`, `steps_distill.py`, `steps_map.py`, `steps_report.py`, `runner.py` (orchestrator).

## Implementation Phases

### Phase 1 (ship first) — No conditions
1. **Exp 6**: Refactor `pipeline/demo_runner.py` into `pipeline/demo/` subpackage
   - Split by step group: `steps_embed.py`, `steps_caption.py`, `steps_ssl.py`, `steps_distill.py`, `steps_map.py`, `steps_report.py`
   - `pipeline/demo_runner.py` becomes thin shim: `from pipeline.demo.runner import run_demo`
   - Async-parallel demo steps: embed + caption can run concurrently; SSL runs after embed completes. Target: <10 min for 58 frames.
2. **Exp 1**: Wire Gemma API as production captioner in `pipeline/indexer.py`
   - `_caption_batch()` prefers GEMMA_API_URL when set; falls back to Florence
   - Parallel chunked requests: validate ollama `--parallel` flag (default 1 = serial; need 3-4 concurrent). If parallel mode supported: 4 concurrent × 3 frames/chunk = 12 frames/wave, ~50s per wave → 58 frames ≈ 5 waves = ≤4 min. If serial only: target degrades to ≤5 min at 16.6s/frame average.
   - `GEMMA_MAX_CAPTION_FRAMES` env var (default 200): cap frames captioned per mission using existing histogram-diff quality ranking to select top-N keyframes. Mission outputs include `caption_model` per frame so operator knows which frames used Gemma vs Florence.
   - No schema changes — stores in existing `caption`, `caption_confidence`, `caption_model` columns
3. **Runbook**: `docs/runbooks/gemma-api.md` — ollama health check, restart, quality validation, Florence fallback procedure

### Phase 2 (conditioned on Exp 3 quality test) — Qwen replacement
- Run 20-frame quality validation: JSON validity ≥ 0.85, vehicle count accuracy ≥ 0.70
- If passes: replace `pipeline/qwen_model.py` HTTP calls with Gemma API equivalents
- If fails: keep Qwen, defer Gemma structured extraction

### Phase 2 (no SSL gate required) — DINOv3→EfficientViT baseline + Qwen quality test
- `models/efficientvit_model.py` + Stage 1→2 distillation from DINOv3 teacher (always proceeds)
- Run Qwen→Gemma quality validation (≥50 frames, gate ≥0.82 JSON validity + ≥0.70 vehicle accuracy)

### Phase 3 (conditioned on SSL gate) — SSL validation + Gemma teacher upgrade
- Build `scripts/validate_ssl_improvement.py`
- Run on ≥3 diverse videos (all required: daylight + low-light + ≥3 GPS waypoints, ≥200 frames each)
- Gate: ΔR@1 > +0.02 on at least 2/3 videos (3 seeds per video; median result used)
- If passes: build `GemmaSSLFinetuner`; then `GemmaVisionTeacher` + Stage 0→1 distillation
- Compare Stage 1→2 quality from Gemma teacher vs DINOv3 baseline (R@1 comparison)
- If gate fails: DINOv3→EfficientViT baseline (Phase 2) is the production edge model

## Decision Trees

**If SSL gate fails (Phase 3):**
→ Skip GemmaSSLFinetuner. Proceed to Stage 1→2 EfficientViT-S1 distillation from DINOv3 teacher (unchanged pipeline). Gemma remains as captioner only.

**If Gemma structured JSON fails Phase 2 quality gate (validity < 0.80):**
→ Keep Qwen for structured extraction. Gemma provides captions only. Re-validate Gemma structured extraction on next model version.

**Gemma API timeout strategy (production indexer):**
Chunked async batch: 3 frames/chunk, 50s timeout, 1 retry per chunk, Florence fallback on 2nd failure. If Florence also fails: store empty string `""` and `caption_confidence=0.5` (existing convention — never NULL). Verify GEMMA_API_URL rate limit before deployment; implement exponential backoff if >10% timeout rate observed. Estimate: 58 frames = ~20 chunks × 50s max = ≤17 min worst case; ≤5 min at 16.6s/frame average.
