## Vision

### 10x Check

The 10x version of captioning isn't "better prose descriptions" — it's an **outdoor
scene intelligence engine**: every keyframe becomes a row in a structured fact table
(objects, counts, positions, road state), queryable with natural language or structured
filters. "Five trucks in a row on a wet road" becomes an exact query, not a semantic
approximation. The diff layer compares scene facts across missions at the same GPS
waypoint, producing human-readable change explanations. The robot advisory API surfaces
scene history during active missions: "the last 3 visits to this junction had wet road
conditions and 3+ trucks." This is the living world model from the original vision —
made real.

### Platonic Ideal

An outdoor robotics engineer drops a mission MP4, waits a few minutes, then types:
"show me every junction where I saw more than 2 vehicles at once, and tell me how the
road condition changed since last week." The system returns exact answers: GPS timestamps,
frame thumbnails, change explanations. The query interface feels like SQL for the real
world. No documentation required. A new engineer on the team can use it on day one.

## Implementation Phases (updated with accepted expansions)

### Phase 1 (ship now) — Florence captioning
- Implement `pipeline/florence_model.py` with `<MORE_DETAILED_CAPTION>`
- Load alongside CLIP/DINOv3 (no sequential unloading — all fit concurrently at 3.4GB)
- Wire into `pipeline/indexer.py` after embed step
- `frame_facts_json JSONB` column added (NULL in Phase 1)
- Caption in Qdrant payload (display-only)
- Backfill script (`scripts/backfill_captions.py`)
- Build eval set: 50-100 real keyframes + 20 target queries

### Phase 2 (conditioned on eval) — Qwen structured facts
- `pipeline/qwen_model.py`: structured JSON extraction
- CLIP pre-screen for vehicle-likely frames: `OpenCLIPTagger.describe_image()` in `pipeline/vision_models.py` (already in codebase). Top label in `{"vehicle","truck","car","bus","convoy"}` with cosine score ≥ 0.25 triggers Qwen. Threshold configurable via `QWEN_CLIP_THRESHOLD` env var (default 0.25). Cost: zero — CLIP embedder already loaded.
- Postgres FTS + GIN indexes (Phase 2 migration, outside transaction)
- Postgres advisory lock for GPU coordination with nerfstudio
- `use_structured_filter` param on existing search endpoint

### Phase 3 — Structured Scene Query API
- `app/routers/scene.py`: `POST /query/scene` endpoint
- Request: `{truck_count_min?, road_condition?, road_surface?, within_radius_m?, top_k?, natural_language?, sort_by?}`
  - `sort_by` defaults to `"recency"` (DESC t_sec); options: `"recency"` | `"embedding_score"`
  - `natural_language` is opt-in per request; if present, Qwen parses it into structured fields.
    Parsed fields are merged with any explicit structural fields (explicit fields take precedence).
    If NL parsing fails or Qwen is unavailable: fallback to vector search, log warning.
- Implementation: JSONB containment + range queries against `frames.frame_facts_json`
- Auth: `X-API-Key` (same as existing endpoints)
- Response: frames matching all filters, ranked by `sort_by`
- Latency target: p99 < 500ms for structured queries; < 3s with NL parsing

### Phase 4 — Semantic Change Detection
- **Additive layer on top of** existing embedding-based change detection in `pipeline/change_detection.py`
  (does NOT replace embedding distance — augments it with semantic explanation)
- For GPS-overlapping frame pairs that already triggered embedding-distance change detection:
  if both frames have non-NULL `frame_facts_json`, compute semantic diff:
  appeared objects, disappeared objects, road condition delta
- Store `semantic_diff_json JSONB` in `change_detections` table (new nullable column)
- Human-readable `change_explanation TEXT` field: "3 trucks present in mission 7, absent in mission 9"
- If either frame lacks `frame_facts_json` (e.g., Phase 2 not yet run): `semantic_diff_json = NULL`, no explanation

### Phase 5 — Scene Timeline per GPS Location
- **Prerequisite: Phase 2 must ship first.** `scene_timeline` is populated from `frame_facts_json` — if Phase 2 eval fails and Qwen is not deployed, this table remains empty. Phase 5 is blocked until Phase 2 ships.
- **Alternative to separate table (YAGNI note):** At current scale, `POST /query/pose` could use a GiST-indexed query on `frames JOIN global_map` directly, without a separate `scene_timeline` table. Accept the separate table only if write throughput or query latency justifies denormalization. Decision: proceed with separate table for clean history semantics; revisit if migration complexity exceeds 1 day.
- New table: `scene_timeline (id, global_map_id, mission_id, frame_id, lat, lon, t_utc, frame_facts_json)`
- Migration: add to `scripts/migrate_postgres.py` as Phase 5 migration; create GiST index on `(lat, lon)` + BTree index on `(global_map_id, mission_id)` for filtering
- Populated after Phase 2 Qwen runs: for each frame with GPS + frame_facts_json, insert row
- `POST /query/pose` extended: return last N scene_timeline entries for the queried GPS location

### Phase 6 — Streaming Caption Updates (RTSP)
- Separate pipeline path: `pipeline/rtsp_captioner.py`
- Subscribes to MediaMTX RTSP stream; buffers frames at caption interval (default: 5s)
- Runs Florence (always) + Qwen (if vehicle-likely) on each buffer frame
- Writes to same `frames` + `frame_facts_json` tables (same schema, live mission context)
- GPU scheduling: uses same Postgres advisory lock scheme introduced in Phase 2
  (`pg_try_advisory_lock(1)`). Florence (small, fast) acquires lock and runs immediately.
  Qwen (large, slow): if lock not acquired within 5s (fast-path skip decision), skip this frame's Qwen pass — log drop. Deferred retry (see `facts_pending` below) runs after lock is released, with up to 60s drain window.
  Expected drop rate during nerfstudio runs: high; add a deferred Qwen queue (same as Phase 2
  `facts_pending_json` mechanism) so dropped frames are retried post-mapping.
  **Deferred retry schema:** add `frames.facts_pending BOOLEAN DEFAULT FALSE` column in Phase 6 migration.
  Set TRUE when Qwen is skipped. Post-mapping worker pass: `WHERE facts_pending = TRUE AND frame_facts_json IS NULL` → run Qwen → set `facts_pending = FALSE`.
- **Qwen VRAM isolation with nerfstudio:** Advisory lock serializes GPU compute, but NOT VRAM residency.
  Qwen must be explicitly unloaded before nerfstudio mapper starts:
  `model.cpu(); del model; torch.cuda.empty_cache()` before releasing lock.
  Qwen is reloaded on demand for the next eligible batch.
- Latency target: Florence caption lag ≤ 10s from RTSP frame capture (soft goal);
  if GPU is shared with nerfstudio, Florence may be delayed by lock wait — document expected
  worst-case: 30s lag when nerfstudio is active on same GPU

## Review Session Decisions (2026-03-27 CEO Review)

### Architecture
- **Florence integration:** Batch post-loop inside `index_video()` (not per-frame). Collect all kept frame PIL images, run Florence in one batched pass after the main loop, update `frame_records` before returning.
- **Florence failure mode:** Hard import, fail fast. Worker crashes if Florence fails to load.
- **Florence `_run_captioning()` helper:** Extract to private method to keep `index_video()` readable.
- **`caption_batch()` return contract:** `List[Tuple[str, float]]` with `len(result) == len(input_frames)` guaranteed. Enforced to prevent silent caption-to-frame misassignment.

### Error handling
- **Qwen JSON parse error:** Whitelist-filter parsed dict to known keys; store `{"parse_error": true, "raw": "<truncated>"}` on failure.
- **Qwen advisory lock release:** Must be in `finally` block. Lock released on exception, not just success.
- **Backfill Qdrant failure:** Log + continue (DB/Qdrant may be temporarily inconsistent; add `--skip-qdrant` flag).
- **GPU OOM at Florence load time:** Call `torch.cuda.empty_cache()` before `model.from_pretrained()`.

### Phase gating
- **Phase 3 gated behind Phase 2** — strict migration order. Phase 3 not deployed until FTS/GIN indexes confirmed created.
- **Phase 5 GPS bucketing:** `SCENE_TIMELINE_RADIUS_M=25m` (configurable). At query time: GiST earth_distance within radius. Schema specifies this before implementation.

### GPU coordination
- **Qwen query-time NL parse:** Uses same `pg_advisory_lock(1)` as index-time Qwen. If lock not acquired within 5s: return 503 `{"error": "nl_parse_busy", "fallback": "use structured params"}`.
- **QWEN_PREWARM=false** default. Document: `p99 < 3s` requires `QWEN_PREWARM=true` on ≥24GB GPU.

### Phase 2 pre-validation (updated — ollama/vLLM sidecar, not AWQ)
- **Sidecar quality gate (replaces AWQ gate):** Stand up ollama Docker service, pull
  `qwen2.5-vl:7b` Q4_K_M (~4.7GB download, ~5.4GB VRAM). Run 20 vehicle frames with
  Phase 2 prompt. Measure: JSON validity ≥ 0.85, vehicle-count accuracy ≥ 0.70,
  p95 latency ≤ 30s, total VRAM ≤ 14GB. Fallback: Q8_0 or vLLM `--cpu-offload-gb 4`.
  **24GB GPU no longer required** — ollama Q4_K_M fits on 16GB GPU alongside Florence.
- **AWQ approach dropped.** Worker calls Qwen via HTTP (OpenAI-compatible API), no
  in-process PyTorch loading, no advisory lock for Qwen VRAM isolation.

### Provenance (NEW — from outside voice)
- **`caption_model` column:** Add `caption_model TEXT` to `frames` table in Phase 1 migration. Store model identifier (e.g., `"florence-2-large"`, `"qwen2.5-vl-7b-awq"`). Prevents silent mixed-model data on model upgrades.

### Phase 6 architectural note (NEW — from outside voice)
- **RTSP live captioning is a stream consumer, not a pipeline file.** `pipeline/rtsp_captioner.py` needs a persistent event loop with RTSP frame buffering — architecturally distinct from the batch job processor. Design separately at Phase 6 time. Do not assume it's a simple addition to the worker.

### Observability
- Structured log events: `florence_captioning` (mission, frames, duration, batch_size, oom_fallback), `qwen_extraction` (frame_id, vehicle_groups count, parse_ok), `gpu_lock_not_acquired` (job, wait_ms, fallback)
- Metric: `caption_null_rate` — percentage of frames with `caption IS NULL` after mission completion
- `/admin/caption-eval` page: shows `caption_null_rate`, mean `caption_confidence`, Precision@5
