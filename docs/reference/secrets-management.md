# Secrets Management

Selfsuvis uses a **layered `.env` model**: each playground has its own
environment variable namespace, and secrets are kept entirely out of version
control via a deliberate file hierarchy.

---

## File hierarchy

```text
/                           <- project root
  .env                      <- generated by ssv-env; gitignored
  .env.example              <- template committed to the repo
  .env.local                <- optional local overrides; gitignored
  .env.secrets              <- production secrets only; gitignored

src/selfsuvis/
  pipeline/core/config/env/ <- core pipeline preset .env files
  sencoop/env/                 <- coop stack preset .env files
  realtime/env/             <- realtime bridge preset .env files
```

The root `.env` is the single source of truth for every process (API, worker,
bridges). Each package ships lightweight preset files that `ssv-env`
merges when generating the root file.

---

## Bootstrap

```bash
# Generates .env from hardware-detected presets
ssv-env

# Inspect what was generated
cat .env
```

`ssv-env` reads the packaged preset files (`core/env/*.env`,
`sencoop/env/*.env`, `realtime/env/*.env`), detects CUDA availability, and writes
the merged output to `.env`. Values already present in `.env` are preserved.

---

## Variable namespaces

| Prefix | Playground | Config class |
|---|---|---|
| *(no prefix or `API_`, `DATABASE_`, …)* | Production server + worker | `Settings` |
| `COOP_` | Coop IoT mesh | `CoopPilotSettings` |
| `REALTIME_` | Realtime SLAM bridges | `RealtimePilotSettings` |

Validate all three at once:

```python
from selfsuvis.config import validate_all
validate_all()
```

## What goes where

### Root `.env` — all runtime config (non-secret defaults OK)

```dotenv
# Core
DATABASE_URL=postgresql://...
API_KEY=change-me
HF_TOKEN=                        # set for private HuggingFace model downloads

# Coop
COOP_MQTT_HOST=localhost
COOP_MQTT_USER=
COOP_MQTT_PASSWORD=              # set when broker requires auth

# Realtime
REALTIME_POSE_BACKEND=stub
```

### `.env.secrets` — production secrets only

Keep the following variables out of `.env` on production hosts and in a
separate `.env.secrets` that is loaded last (or set via the host's secret
manager):

| Variable | Risk if leaked |
|---|---|
| `API_KEY` | Full API access |
| `DATABASE_URL` | Full DB access including all indexed video metadata |
| `HF_TOKEN` | HuggingFace account access, private model download |
| `CVAT_WEBHOOK_SECRET` | CVAT annotation webhook spoofing |
| `COOP_MQTT_PASSWORD` | MQTT broker authentication |
| `REALTIME_JWT_SECRET` | Realtime session token forgery |

Load order when using a secrets file:

```bash
# shell init or Docker entrypoint
set -a
[ -f .env ] && . .env
[ -f .env.secrets ] && . .env.secrets
set +a
```

Docker Compose example:

```yaml
services:
  api:
    env_file:
      - .env
      - .env.secrets      # overrides .env values; mount from secret store
```

## Coop stack secrets

The sencoop stack uses three additional secret stores managed outside Python:

| Path | Purpose |
|---|---|
| `config/sencoop/mosquitto/certs/` | TLS certificates for MQTT broker |
| `config/sencoop/chirpstack/certs/` | TLS certificates for ChirpStack |
| `config/sencoop/*/passwords` | Hashed broker/service passwords |
| `config/sencoop/*/.secrets` | Per-service secret files |

All of these paths are gitignored. Generate them with:

```bash
scripts/sencoop/sencoop-bootstrap.sh
```

Set `COOP_MQTT_TLS=true` and supply `COOP_MQTT_USER` / `COOP_MQTT_PASSWORD`
when the broker requires authenticated TLS connections.

## Realtime bridge secrets

Realtime bridge sessions can optionally authenticate via a shared JWT secret:

```dotenv
REALTIME_JWT_SECRET=             # leave blank to disable JWT auth on bridges
```

When non-empty, the bridge runtime validates tokens before accepting pose or
occupancy streams.

---

## Secret rotation

1. Generate a new value (e.g. `openssl rand -hex 32`).
2. Update `.env.secrets` (or the host secret manager entry).
3. Restart the affected service: `docker compose restart api worker`.
4. For `DATABASE_URL` rotation, also restart `mapper` and any bridge runtimes.

There is no in-process hot reload of secrets. All secrets are read once at
startup via `python-dotenv` / pydantic-settings.

---

## Startup validation

The worker and API both call `validate_settings()` at startup, which will log
a warning if `API_KEY` is the default placeholder and raise a `ValueError` for
hard misconfigurations (wrong backend names, out-of-range window sizes, etc.).

For stricter enforcement set:

```dotenv
STARTUP_PREFLIGHT_STRICT=true
```

This causes the process to exit rather than warn if critical services are
unreachable at startup.
